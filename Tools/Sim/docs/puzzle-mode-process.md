# Puzzle Mode — Process, Attempts, and Struggles (for review)

A candid write-up of how the OPTCG "lethal puzzle" mode was built, everything attempted, the dead ends, and
the open problems. Written for an external reviewer (Codex). Nothing here is polished marketing — it's the
real path, including what's shaky.

## 0. Goal

> Superseded for current implementation: see `puzzle-mode-v2.md` for the mechanic-graph, cross-turn proof
> architecture. The history below is retained as a record of the approaches that failed.

A chess-puzzle-style solo mode: **one board, your turn, a guaranteed forced win exists — find the line.** Not
an author's scripted sequence: the mode accepts ANY line the engine proves wins against optimal defense.
Difficulty from "swing with everything" up to tight, resource-constrained, counter-aware lines.

## 1. Architecture (files)

All under `Assets/Scripts/Engine/Puzzles/` (shared: compiled by both Unity and the headless `Tools/Sim`):

- **`LethalSolver.cs`** — the core. Exact one-turn **AND/OR minimax** over the REAL engine (clone + ApplyCommand
  via `Bot.Search.GameClone`/`LegalActions`). Returns `Win | NoLethal | Unknown` with a work-budget counter and
  a principal variation. `DecidingSeat`, `StrongestDefense` (the adversary: refute if possible, else make the
  attacker work hardest). Move ordering puts setup moves first so a win is found fast.
- **`HintGenerator.cs`** — 3 graduated hints derived from the solved PV (never points off-solution): L1 theme
  nudge, L2 names the key card, L3 spells the first moves. `DescribeSolution` reveals the full line on strike-out.
- **`PuzzleRuntime.cs`** — the play loop. Player controls the attacker; `ApplyMove` applies the move then
  auto-plays the defender's `StrongestDefense` and advances to the next player decision / turn-end / game-end.
- **`PuzzleGenerator.cs`** — procedural generator, 7 families (DonPush, BlockerBreak, CounterPump, ComboWall,
  DoubleStrike, RushFinisher, DoubleWall). Deterministic `xorshift64` RNG so a seed reproduces byte-for-byte in
  Unity AND the Sim.
- **`PuzzlePalette.cs`** — AUTO-GENERATED per-color legal card pools (leaders, vanilla bodies, blockers,
  counters, events, removal, DA, Rush, life). Generated by `$LOCALAPPDATA/Temp/claude/emit_palette.py`.
- **`PuzzleLibrary.cs`** — `VerifiedSeeds` (the certified generated puzzles) + `AuthoredPuzzle` (a `Func<GameState>
  Build` + metadata).
- **`Mate2Solver.cs`** — a two-ply solver (see §5, abandoned as a puzzle family).
- **`HarvestedPuzzle.cs` / `DifficultyEvaluator.cs`** — real-game harvesting types (see §7).
- **`Tools/Sim/Puzzles/PuzzleHarvester.cs`** — the harvester (Sim-only).
- Unity glue: `Assets/Scripts/GameManager.Puzzle.cs`, plus preview/UI in `GameManager.cs`.

Diagnostic commands in `Tools/Sim/Program.cs`: `lethaltest`, `puzzlegen` (certifier), `puzzlecheck`
(end-to-end solve of every baked puzzle), `puzzleharvest`, `harvestcheck`, `odentest`, `vivitest`, `datest`.

**Headless gate:** the shipping engine is pure C#, so `Tools/Sim` compiles it and runs the solver/runtime with
zero Unity. This is the ONLY way I could validate anything — Unity MCP is entitlement-blocked, so every
Unity-side file gets its first compile on the user's build.

## 2. Generated puzzles — approach and the "too easy" struggle

The generator builds a random board per family, and a **certifier** (`puzzlegen`) proves each seed is exactly
forced-lethal, drops any that a plain swing wins ("swing-only"), buckets by difficulty, and I re-bake the
surviving seeds into `PuzzleLibrary.VerifiedSeeds`.

**Struggles / iterations (all real user feedback):**
- First pass: "none of these are even remotely hard — no thought." The generator produced boards where you
  just swing. Fix: the swing-only filter in the certifier + families that REQUIRE setup (play a piece, break a
  blocker, spread DON, play around counters).
- "The bot has counters and DON but isn't using them — no interactions." → `StrongestDefense` was tuned to
  defend optimally to the end (block + counter to delay), not pass.
- "Cards must be color-legal for the leader." → the palette is per-color; the generator only draws legal cards.
- "Only using the same leader — need variety." → `Blank()` sets a random leader; rich defensive leaders (Nami,
  Doflamingo, Teach, etc.) included.
- Even after all that: **"expert isn't hard."** This is the core limitation — the generator produces
  *combinatorially* varied but *strategically shallow* boards. It can't produce the tight, multi-constraint
  lines that real games do. That's what pushed me to real-game harvesting (§7).

### Palette data reliability — a recurring trap
The palette is emitted from the scraped card DB, and the DB is messy:
- **Multi-record cards:** e.g. EB01-023 has 3 records (first=8000/counter1000, alt-arts=6000/counter0). The
  engine's loader is first-record-wins; my emit script initially matched a later record → wrong stats → a
  "counter" puzzle that wasn't. Fix: dedupe FIRST-record-per-id to mirror the engine.
- **Conditional blockers:** "If … gains [Blocker]" matched a naive `[Blocker]` search → fake walls. Fix: innate
  only (`effect.lstrip().startswith('[Blocker]')`).
- **Counter-0 life:** don-push puzzles failed at ~60 work because stripped life cards carried counter values the
  engine treats as counters. Fix: filter life to `counter==0 and trigger=='' and no [Counter]`.
- **Removal parser bug:** OP12-029's "Rest … Then, K.O. up to 1 rested Character …" hits an engine parser gap
  ("unknown condition 'your rested character would be K.O.'d'"). Fix: restrict palette removal to single
  `[On Play]` clauses (dodges the engine bug rather than fixing the interpreter).

**Lesson:** every palette change forces a re-certify (`puzzlegen`, ~2.5 min) + re-bake of `VerifiedSeeds`.
This churn is real friction; the seeds are only valid against the exact engine + palette they were certified on.

## 3. The resolution-UI struggle (rich leaders / `[When Attacking]`)

User: "I like the leader abilities, but puzzle mode doesn't give the resolve options PvP does." Attacking with
a `[When Attacking]` leader (Arlong/Katakuri/Sakazuki) left a pending effect the puzzle UI never prompted → the
attack hung.

Root cause: `DrawContextActions` short-circuited to `DrawPuzzleActions` (the status/hints panel) BEFORE the
normal pending-effect UI. Fix: in puzzle mode fall through to the real resolution UI whenever the PLAYER has a
pending decision (pending effect / choice / deck-look / char-replace); only show the puzzle panel when idle. The
runtime already handled these headlessly (puzzlecheck proves it), so it was purely a UI gate. Added a
direct-apply fallback in `ApplyMove` for resolution commands whose exact target `LegalActions` doesn't enumerate
(its target set is pruned for search) — apply straight through the engine like PvP.

**Reframe that mattered:** my first instinct was to STRIP the rich leaders from the palette. The user correcting
me ("I like the abilities") flipped it to fixing the UI instead. Keeping the depth was the right call.

## 4. Hide opponent hand

`BottomSeat` now = `puzzleAttacker` (player always at the bottom), and the opponent's top hand renders face-down,
which also disables its hover preview. Face-down Life remains hidden; only a `CardInstance` explicitly marked
`FaceUp` renders and previews its front. The solver still has full state; the player doesn't.

## 5. Strike system + no-lethal detection (and the abandoned mate-in-2)

- **Mate2Solver:** I built a two-ply solver to make "mate in 2" puzzles. It PROVED the family infeasible for
  normal lethal — the defender can always DON-boost / block to survive a turn, so a forced two-turn mate almost
  never exists in constructed boards. Abandoned as a family; kept the file as a foundation.
- **Strike system:** a strike is counted when an attempt fails. Two detections: (a) `OnlyEndTurnLeft` (every
  attacker spent, only endTurn legal) and (b) after each move, RE-SOLVE and fail immediately if the position is
  PROVEN `NoLethal` (not merely `Unknown` within budget). (b) covers "all attackers spent but I still hold DON +
  a non-Rush card → no way to win" — flag it now, not after the dead card is played. It does NOT leak the
  answer: a still-winning position returns `Win` and stays silent. 3 strikes → reveal the line.

## 6. Difficulty reshuffle

User judged generated puzzles too easy even at the top. So: **drop the old generated Easy/Medium entirely; keep
only old Hard/Expert relabelled DOWN two tiers (Hard→Easy, Expert→Medium).** The genuinely Hard/Expert tiers are
owned by the real-game HARVESTED puzzles. Done as a display-level remap in `PuzzleLibrary.All()` (`meta.Difficulty
< 3` dropped; else `-2`), so the generator/certifier are untouched.

## 7. Real-game harvesting — the biggest effort and the biggest struggles

**Idea:** turn actual recorded games into puzzles. Every game already saves `{Seed, deck ids, CommandHistory}`
(ReplayStore) and per-turn snapshots (GameLogStore). Reconstruct the exact position and, at each decision point
where the active player is on-move in their main phase, run the LethalSolver; every proven-lethal position is a
candidate. Score difficulty, emit the hardest.

**Design decisions (user-chosen):**
- **Exact replay reconstruction** (not snapshots): `CreateMatch(Seed, deck defs) + replay CommandHistory`. Gives
  EXACT mid-turn states (DON partly spent, a character already played) as well as turn-starts — the
  resource-variety the user wanted. Requires both deck lists (starter → `CardData.StarterDecks`, custom →
  `persistentDataPath/Decks/*/decks.json`).
- **Worst-case triggers = free:** because reconstruction is exact, the defender's real face-down Life and deck
  order are present, so the solver's adversarial search already accounts for disruptive `[Trigger]`s. No special
  handling needed.
- **Storage:** a self-contained JSON recipe (seed + both deck lists + command prefix) in
  `StreamingAssets/Puzzles/harvested.json`, reloaded by `HarvestedPuzzle.Build()` (CreateMatch + replay). The
  same JSON round-trips Unity `JsonUtility` (client) and `System.Text.Json` (Sim).

**What worked:** reconstruction is SOLID — all 22 local replays replayed cleanly (instance IDs are
deck-order-deterministic, so exact decks reproduce them). `harvestcheck` confirms every emitted recipe rebuilds
a valid forced-lethal position.

**The struggles (this is the important part):**
1. **Yield is genuinely low.** Real casual-game wins are rarely FORCED lethals vs optimal defense — the loser
   usually just ran out of resources or misplayed, not "no defense could survive." Across 22 games I found only
   ~8–12 forced-lethal positions total. Most decision points come back `NoLethal`.
2. **The solver is slow on real boards.** Synthetic puzzles have 2–3 attackers; real end-game boards have
   5 characters + leader + DON + counters, so branching is factorial. A single solve at a useful budget can take
   many seconds; a game with U11/U13 (11–13 `Unknown` positions where the budget was exhausted) could take
   minutes. Budget tuning is a genuine tension: too low misses real lethals (`Unknown`), too high stalls.
3. **Budget/filter interactions are finicky.** Adding a `defLife<=3` pre-filter to cut solves changed WHICH
   decision points got solved, and a per-game solve cap changed it again — so the SAME games yielded different
   counts under different settings. Hard to get a stable, complete harvest.
4. **The emit-only-at-end architecture bit me repeatedly.** The harvester emits the JSON only after processing
   ALL games. The last few games (branchy, slow) kept stalling, and every time I killed the run to re-tune, I
   LOST the in-memory candidates from the fast games — nothing was written. I burned several cycles this way.
   **This is the single biggest process failure and the clearest thing to fix: emit incrementally (or
   checkpoint), so a slow tail can't discard good early results, and so runs are resumable.**
5. **Resolution:** I ended up pointing the harvester at ONLY the games known to yield (copy the 4 yielding
   replays into a temp base), which completed in ~2 min and emitted 4 Expert puzzles. So the shipped harvested
   set is small (4), all rated Expert by the evaluator (big boards + setup + counters score high).

**DifficultyEvaluator:** scores a proven-lethal position on setup-plays-before-swing (×3), activate-first (×2.2),
DON allocation, defender counters-in-hand + blockers, extra swings over the minimum, and solver work; buckets
1–4. Not empirically validated against human difficulty — it's a reasonable heuristic, not tuned.

## 8. Engine bugs the puzzles surfaced (fixed)

Playing real leaders in puzzles exposed several ENGINE bugs (the engine is a brittle substring interpreter):
- **Oden EB01-001** `[DON!! x1] [When Attacking]` buff fired with 0 DON — `ExtractTimedClause` started the clause
  AT the timing tag and DROPPED the leading `[DON!! xN]` gate (standard print order is `[DON!! xN] [When
  Attacking]`), so `ParseDonThreshold` read 0. Same drop silently disabled On-Block / Activate:Main DON gates.
  Fixed at the chokepoint (retain leading bracket tags contiguous with the timing tag).
- **Vivi EB03-001** fired a phantom `[When Attacking]` every swing — `HasTiming` matched the INLINE reference
  "…Characters without a **[When Attacking]** effect…". Fixed: `HasTiming` only counts a tag in a line's leading
  bracket-tag run, not mid-sentence.
- **Double Attack defeat rule** — the engine let `[Double Attack]` win against a Leader at 1 Life. Official Q36:
  it does NOT (the overkill of a single hit doesn't defeat). I initially asserted the WRONG rule from reasoning
  and dismissed the bug report — corrected after the user cited the official Q&A. Fixed so only a hit landing on
  a 0-Life Leader defeats. **Lesson recorded: verify OPTCG rules against the official Q&A, never from reasoning.**
- **Perona OP06-021** `[Activate: Main]` couldn't be used in puzzles — the puzzle panel replaced the action
  window before the selected-card branch that shows the Activate button. Fixed: in puzzle mode fall through to
  the normal action UI whenever the player has a selection.

## 9. Open problems / for review

1. **Possible false-lethal (Iron Body, unresolved).** A playtest report: the puzzle's auto-defender tapped 2 DON
   but SKIPPED a counter event (OP07-095 Iron Body, +4000/+6000 at 10+ trash) that "could have let it survive."
   If the solver / `StrongestDefense` does not fully model the defender PLAYING counter EVENTS (as opposed to
   counter cards / blocking), some generated puzzles that include counter events could be "lethal" only because
   the defense is undervalued. I did NOT get to verify whether `LegalActions` enumerates counter-event plays for
   the defender in the solve. **This is the highest-priority correctness item to audit.**
2. **Harvester architecture:** emit incrementally / make runs resumable (see §7.4). Also worth: a cheaper
   pre-filter (necessary-condition bound on max life damage) before the expensive solve, and a two-tier solve
   (cheap reject → deep only on the ambiguous) done RIGHT (my two-tier attempt at 180k was too slow).
3. **Solver performance on big boards** — the factorial branching (attack order × DON allocation × targets) is
   the bottleneck. Better move ordering / transposition / pruning would raise both harvest yield and speed.
4. **DifficultyEvaluator is unvalidated** — no ground truth for the buckets.
5. **Certifier/palette churn** — `VerifiedSeeds` are only valid for the exact engine+palette they were certified
   against; any engine change (like the DA fix) can silently invalidate baked puzzles. `puzzlecheck` is the guard
   but it's a re-solve of all 50, not a proof of stability.
6. **Two oracles** — `IsAutomatedEffectPattern` (auto-resolve gate) ⊂ `TryResolveKnownEffect` (real resolver).
   The runtime uses the real engine so it's consistent, but the generator's swing-only certification assumes the
   engine's power/rules are correct — any engine bug (§8) can produce a "certified" puzzle that's actually wrong
   until the bug is fixed and it's re-certified.

## 10. Meta-lessons

- **The headless Sim gate is everything.** Every engine/runtime claim was proven in `Tools/Sim` (lethaltest,
  puzzlecheck, smoke). I cannot compile Unity, so UI code is unverified until the user builds.
- **Puzzles are a fuzzing harness for the engine.** Dropping real leaders onto boards and demanding a proven
  lethal surfaced four distinct engine bugs the normal game never hit. That's a side benefit worth keeping.
- **Don't strip depth to dodge a bug; fix the substrate.** Twice my first instinct was to remove hard cases
  (rich leaders; multi-clause removal). The rich-leader case was better solved by fixing the UI/engine; the
  removal case I DID dodge (single-clause palette) because the underlying parser bug was out of scope.
- **Verify rules against the source.** The Double Attack episode is the clearest failure of reasoning-from-
  intuition on a counter-intuitive TCG ruling.
